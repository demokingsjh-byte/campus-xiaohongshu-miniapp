# 云点校园人员配置方案

## 1. 配置原则

项目属于“后台管理 + 小程序 + 多租户后端 + 交易分润”的综合型产品，不能只按普通小程序配置人员。

人员配置建议分成两个版本：

- 精简 MVP 团队：适合先做可演示、可试点版本。
- 标准商业化团队：适合进入支付、分润、商家、运营增长阶段。

## 2. 精简 MVP 团队

适合周期：

```text
6 到 8 周
```

适合目标：

- 跑通后台基础管理。
- 跑通小程序核心页面。
- 跑通用户、校区、内容、商品基础链路。
- 能部署到服务器演示。

### 人员配置

| 角色 | 人数 | 主要职责 |
| --- | ---: | --- |
| 产品负责人 | 1 | 需求拆解、原型确认、验收标准、排期协调 |
| 后端开发 | 1 | 数据库设计、接口开发、租户隔离、后台 CRUD、部署 |
| 前端开发 | 1 | 后台前端改造、小程序页面、接口联调 |
| UI/视觉 | 0.5 | 小程序视觉、图标、页面风格、基础设计规范 |
| 测试/实施 | 0.5 | 功能测试、真机测试、部署验证、问题记录 |

合计：

```text
4 人左右
```

其中 UI 和测试可以兼职，但不能完全没有。

## 3. 标准商业化团队

适合周期：

```text
10 到 16 周
```

适合目标：

- 上线真实小程序。
- 接入微信登录、支付、退款。
- 完成订单和分润。
- 支持代理人运营。
- 支持多校区试点。

### 人员配置

| 角色 | 人数 | 主要职责 |
| --- | ---: | --- |
| 产品经理 | 1 | 业务流程、版本规划、需求验收、运营策略 |
| 项目经理 | 0.5 | 进度跟踪、风险管理、跨角色协调 |
| 后端开发 | 2 | 业务接口、订单支付、分润、后台管理、定时任务 |
| 后台前端开发 | 1 | 管理后台页面、权限菜单、表单、数据看板 |
| 小程序开发 | 1 到 2 | 小程序首页、发布、交易、登录、用户中心 |
| UI/交互设计 | 1 | 小程序视觉、后台关键页面、交互规范 |
| 测试工程师 | 1 | 功能测试、接口测试、兼容性测试、回归测试 |
| 运维/DevOps | 0.5 | 服务器、数据库、Redis、CI/CD、日志、备份 |
| 运营人员 | 1 | 校区资料、种子内容、代理人沟通、试点反馈 |

合计：

```text
8 到 10 人
```

## 4. 各岗位详细职责

### 产品经理

职责：

- 明确 MVP 边界。
- 维护需求文档。
- 梳理用户流程。
- 梳理后台管理流程。
- 确认字段、状态、权限。
- 定义验收标准。
- 组织每周版本评审。

重点产出：

- 产品需求文档
- 页面原型
- 功能清单
- 验收用例
- 版本计划

### 后端开发

职责：

- 数据库建模。
- 多租户隔离。
- 用户模块。
- 区域校区代理模块。
- 内容模块。
- 商品模块。
- 订单模块。
- 分润模块。
- 文件上传 OSS。
- 小程序 API。
- 后台管理 API。

重点产出：

- SQL 脚本
- 后端接口
- Swagger/API 文档
- 后台 CRUD
- 部署配置

### 后台前端开发

职责：

- 后台菜单整理。
- 区域管理页面。
- 校区管理页面。
- 代理人管理页面。
- 内容管理页面。
- 商品管理页面。
- 订单管理页面。
- 分润管理页面。
- 数据看板。

重点产出：

- 可用后台页面
- 表格筛选
- 新增编辑弹窗
- 审核操作
- 数据导出入口

### 小程序开发

职责：

- 校区选择。
- 首页内容流。
- 内容详情。
- 发布闲置。
- 发布互助。
- 用户登录。
- 我的发布。
- 我的交易。
- 消息入口。
- 分享邀请。

重点产出：

- 微信小程序可运行包
- H5 预览版本
- 小程序接口联调
- 真机兼容修复

### UI/交互设计

职责：

- 小程序首页视觉。
- 内容卡片样式。
- 发布流程。
- 我的页面。
- 校区选择页。
- 图标和空状态。
- 基础色彩和组件规范。

重点产出：

- Figma/即时设计稿
- 页面标注
- 图标资源
- 交互说明

### 测试工程师

职责：

- 功能测试。
- 接口测试。
- 小程序真机测试。
- 后台权限测试。
- 多租户数据隔离测试。
- 支付和分润回归测试。

重点产出：

- 测试用例
- Bug 清单
- 回归报告
- 上线检查表

### 运维/DevOps

职责：

- 服务器环境。
- MySQL、Redis、Nginx。
- GitHub Actions。
- 自动部署。
- 日志查看。
- 数据备份。
- 域名和 HTTPS。

重点产出：

- 部署手册
- 备份方案
- 监控方案
- 回滚方案

### 运营人员

职责：

- 收集校区资料。
- 维护初始内容。
- 对接代理人。
- 组织种子用户。
- 反馈产品问题。
- 制定试点规则。

重点产出：

- 校区资料表
- 代理人名单
- 种子内容清单
- 试点反馈报告

## 5. 当前阶段推荐配置

当前处于 MVP 初期，推荐：

| 角色 | 建议 |
| --- | --- |
| 产品负责人 | 必须有，需求要有人拍板 |
| 后端开发 | 必须有，优先保证租户和基础模块 |
| 小程序/前端开发 | 必须有，先做可演示页面和联调 |
| UI | 可以兼职，但至少要统一小程序风格 |
| 测试 | 可以兼职，但每次部署必须验收 |
| 运维 | 可以由后端兼任，但必须保留部署文档 |

最小可执行配置：

```text
产品 1 人 + 后端 1 人 + 前端 1 人 + 测试/运营 1 人
```

## 6. 不建议的配置

不建议只有一个全栈开发长期单人推进。

原因：

- 后台、后端、小程序、部署、数据库、支付分润都需要关注。
- 单人可以搭骨架，但进入交易和分润后风险会快速上升。
- 没有产品和测试，需求容易不断变形。
- 没有运营参与，校区代理模型很难验证。

## 7. 招人优先级

如果只能逐步加人，优先级建议：

1. 后端开发
2. 小程序开发
3. 产品/运营
4. UI 设计
5. 测试
6. 运维

其中后端和小程序是最硬的产能瓶颈，产品/运营决定方向是否正确。
