# 区域校园小红书小程序项目规划

## 1. 项目定位

本项目是一个按校区独立运营的校园综合服务小程序，核心形态类似“校园版小红书 + 校园交易平台 + 校区代理运营系统”。

平台不是单纯的信息发布工具，而是一个具备区域代理、校区租户、校园内容、闲置交易、商家入驻、支付分账、裂变推广能力的校园本地生活平台。

## 2. 技术主线

### 后台与服务端

采用 RuoYi-Vue-Pro / Yudao 作为成熟后台底座，复用：

- 用户、角色、菜单、权限
- 登录、JWT、Redis、Spring Security
- 操作日志、登录日志
- 字典、参数配置
- 代码生成器
- 多租户基础能力
- 支付、商城等可选模块基础

目录：

```text
campus-platform/
```

### 小程序端

采用 Uni-app Vue3 / Vite / TypeScript 模板，复用：

- 请求封装
- Pinia 状态管理
- 路由拦截基础
- 环境配置
- 小程序/H5 多端构建

目录：

```text
campus-miniapp/
```

## 3. 核心业务模型

### 3.1 租户模型

本项目的核心不是“学校字段”，而是“校区租户”。

```text
平台
-> 区域
-> 校区租户
-> 校区代理人
-> 用户 / 商品 / 订单 / 内容 / 商家 / 分润
```

RuoYi 自带的 `system_tenant` 作为 SaaS 租户底座。校园项目通过 `campus_tenant_profile` 给每个租户补充校区业务资料。

### 3.2 校区代理

后台创建一个校区后，可以分配代理人。代理人负责该校区运营，并参与后续分润、裂变、用户增长、商家拓展。

关键表：

```text
campus_tenant_profile
campus_agent
campus_user_tenant
campus_invite_relation
campus_commission_rule
campus_commission_record
```

### 3.3 请求租户识别

小程序端用户进入某个校区后，请求统一携带：

```text
X-Tenant-Id: 租户ID
tenant-id: 租户ID
```

后端根据租户 ID 做数据隔离、权限判断、订单归属和分润计算。

## 5. 后台优先级

后台先行，因为后台决定数据模型、租户、权限和运营边界。

后台开发顺序：

1. 跑通 RuoYi 原始系统。
2. 执行 `campus-extension.sql`。
3. 使用代码生成器生成校园运营菜单。
4. 完成区域管理。
5. 完成学校资料管理。
6. 完成校区管理，并绑定 `system_tenant`。
7. 完成代理人管理。
8. 完成分润规则管理。
9. 完成商品审核管理。
10. 完成用户与校区关系管理。

## 6. 小程序优先级

小程序开发顺序：

1. 环境配置。
2. 校区选择页。
3. 当前校区缓存。
4. 微信登录。
5. 首页信息流。
6. 闲置商品列表。
7. 发布闲置。
8. 商品详情。
9. 我的页面。
10. 订单页面。

## 7. 排期规划

以下按 1 名后端 + 1 名前端 + 1 名测试/产品协同估算。如果只有 1 人开发，时间需要适当翻倍。

### 第 1 周：框架落地与数据库初始化

目标：框架能跑，数据库能用。

交付：

- RuoYi 后台启动
- 云数据库初始化
- 执行 RuoYi 原始 SQL
- 执行校园扩展 SQL
- 小程序模板构建通过
- 确认开发、测试、生产环境配置方式

验收：

- 后台能登录
- 数据库表完整
- 小程序能编译微信小程序产物

### 第 2 周：后台校园运营基础

目标：后台可以维护区域、学校、校区。

交付：

- 区域管理 CRUD
- 学校资料 CRUD
- 校区租户资料 CRUD
- 校区和 RuoYi 租户绑定逻辑

验收：

- 后台能创建区域
- 后台能创建校区
- 校区能绑定或生成租户

### 第 3 周：代理人与分润规则

目标：校区可以分配代理人，并配置分润。

交付：

- 代理人管理
- 代理邀请码
- 分润规则管理

验收：

- 每个校区可指定主代理
- 每个代理有唯一邀请码
- 每个校区可设置交易分润比例

### 第 4 周：小程序校区与登录

目标：用户进入小程序后能归属校区。

交付：

- 当前校区缓存

验收：

- 小程序请求能带 `X-Tenant-Id`
- 用户登录后有当前校区

### 第 5-6 周：闲置交易 MVP

目标：跑通发布、审核、浏览、下单。

交付：

- 商品审核

- 商品详情

验收：

## 9. 风险与注意事项

1. 不要把校区只做成 `school_id` 字段，必须走租户模型。
2. 历史订单必须固化代理人、邀请人和分润规则快照。
3. RuoYi 底层模块名 `yudao-*` 暂时保留，避免破坏框架依赖。
4. 业务表使用 `campus_*` 命名，便于和框架表隔离。
5. 微信支付、分账、提现涉及资金合规，必须先用测试环境验证。
6. 后台菜单和代码生成优先，避免手写大量重复 CRUD。
